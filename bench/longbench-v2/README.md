# LongBench v2

This harness runs Lash against LongBench-style datasets and writes prediction files in the JSONL shape expected by the upstream LongBench evaluation flow.

It is intentionally lightweight:

- one `lash -p` call per example
- input assembled from `context` + `input`
- outputs written under ignored `.benchmarks/longbench-v2/`
- per-dataset prediction files emitted in LongBench-compatible JSONL form

The harness is designed to work with the upstream `THUDM/LongBench` repository and datasets, without baking those artifacts into this repo.

## Expected dataset shape

The runner expects JSON or JSONL rows with the standard LongBench-style fields:

- `input`
- `context`
- `answers` (list)
- `dataset`
- `all_classes` (optional)
- `length` (optional)
- `_id` (optional)
- `language` (optional)

## Quickstart

```bash
bench/longbench-v2/setup.sh
bench/longbench-v2/run.sh --dataset-path /path/to/longbench.jsonl --limit 10
```

Or download lazily into ignored benchmark state:

```bash
bench/longbench-v2/run.sh --dataset-url https://example.com/longbench.jsonl --limit 10
```

Or use an encoded LongBench preset:

```bash
bench/longbench-v2/run.sh --dataset-preset narrativeqa --limit 10
```

If your dataset contains multiple `dataset` names, the harness will emit one prediction file per dataset.

## Output layout

Everything generated by the harness goes under:

```text
.benchmarks/longbench-v2/
```

Each run gets a timestamped directory containing:

- `manifest.json`
- `results.json`
- `pred/<run-name>/<dataset>.jsonl`

Each JSONL line is shaped for the upstream evaluator:

```json
{"pred":"...", "answers":["..."], "all_classes":null, "length":1234}
```

## Evaluate with upstream LongBench

Clone the upstream repo once via setup:

```bash
bench/longbench-v2/setup.sh
```

Then either:

- copy or symlink the generated dataset JSONL files into the upstream `pred/<model-name>/` directory, or
- use `bench/longbench-v2/evaluate.sh` to do that wiring for you.

Example:

```bash
bench/longbench-v2/evaluate.sh .benchmarks/longbench-v2/runs/<run-id>
```

## Useful flags

```bash
bench/longbench-v2/run.sh --dataset-path /path/to/longbench.jsonl
bench/longbench-v2/run.sh --dataset-preset qasper --limit 20
bench/longbench-v2/run.sh --dataset-url https://example.com/longbench.jsonl --limit 20
bench/longbench-v2/run.sh --dataset-path /path/to/longbench.jsonl --dataset-name narrativeqa
bench/longbench-v2/run.sh --dataset-path /path/to/longbench.jsonl --model google/gemini-2.5-pro-preview
bench/longbench-v2/run.sh --dataset-path /path/to/longbench.jsonl --execution-mode rlm
bench/longbench-v2/run.sh --dataset-path /path/to/longbench.jsonl --execution-mode standard --context-approach rolling_history
bench/longbench-v2/run.sh --dataset-path /path/to/longbench.jsonl --max-concurrency 4
```

## Notes

- This first version is a benchmark harness, not a research reproduction of custom KV-cache methods.
- It uses `lash` CLI autonomous mode for simplicity and reproducibility.
- API credentials are read from `.env` if present.
- Lazily downloaded datasets are stored under ignored `.benchmarks/longbench-v2/data/`.
- `--dataset-preset` downloads and extracts the upstream LongBench `data.zip` archive on first use, then reuses the local extracted dataset file.
- Upstream LongBench artifacts are cloned into ignored benchmark state under `.benchmarks/longbench-v2/vendor/LongBench`.
